# 高难问题清单

本页不提供标准答案，只收集那些真正值得反复追问的问题。

如果你准备长期理解这个项目，而不是只会运行脚本，那么这些问题应该被反复拿出来审视。它们多数没有一步到位的答案，有些甚至会随着实验结果变化而被重写。

## 1. 规则与问题建模

1. 为什么六洲棋适合 AlphaZero 风格路线，而不是优先做大规模监督学习或规则启发式系统？
2. 六洲棋的阶段化动作语义，是否天然破坏了“统一策略分布”的假设？
3. 原子阶段拆分到底是在帮助学习，还是在把环境建模复杂性转移到动作空间设计里？
4. 如果把多个原子动作重新合并成更粗粒度动作，训练会更容易还是更难？为什么？
5. 当前规则中的 `MARK_SELECTION`、`CAPTURE_SELECTION`、`FORCED_REMOVAL` 是否在学习难度上存在本质不对称？
6. 对模型来说，“方”和“洲”是应当由网络自己发现的结构，还是应当通过更强的归纳偏置显式注入？
7. 当前规则里“不允许优先移除特殊结构内棋子”的约束，会不会让策略学习产生强烈的局部不连续性？
8. 平局保护规则是工程需要还是任务定义的一部分？如果删掉它们，训练目标会变得更真实吗，还是更不可学？
9. 为什么 `NO_CAPTURE_DRAW_LIMIT=36` 是合理的？它是在逼迫有效对局，还是在扭曲真实博弈分布？
10. 六洲棋的长期信用分配难点，主要来自规则复杂度、动作依赖性，还是来自平局主导分布？

## 2. 状态表示与动作表示

1. 当前 `11` 通道输入真的足以表达所有关键决策信息吗？
2. 阶段 one-hot 是必要信息，还是在替网络“偷懒”提供语义提示？
3. 如果移除 phase one-hot，只保留棋盘与标记信息，模型是否仍能隐式恢复当前阶段？
4. 目前三个策略头的拆分方式，是源于动作语义本身，还是只是为了实现方便？
5. 为什么移动动作建模成 `from-head + to-head` 组合是合理的？这种分解会不会错误地假设动作概率可加性？
6. 三头策略输出是否真的优于一个统一 flat action head？如果优于，优势来自表达能力、优化稳定性，还是样本效率？
7. 动作编码与规则阶段绑定得这么紧，会不会导致未来规则微调时整个训练资产贬值？
8. 当前 flat action spec 是否存在语义碰撞或学习不均衡问题？
9. `placement / movement / selection / auxiliary` 的动作密度差异，会不会让 softmax 学习天然偏向某些阶段？
10. 如果把特殊动作也张量化成更强结构先验，是否能减少策略头在稀有动作上的学习困难？

## 3. MCTS 设计

1. 当前项目为什么仍然坚持 MCTS，而不是直接做纯 policy/value 推理 agent？
2. MCTS 在六洲棋里到底解决了什么问题，是改善探索、改善 target 质量，还是只是提供更稳的 early training bootstrap？
3. 当训练后期模型已经较强时，MCTS 的边际收益是否在下降？如果是，下降发生在哪个环节？
4. 当前根节点搜索得到的 `pi`，真的比原始 policy 更接近“应学目标”吗？
5. 当 MCTS 与价值头意见相左时，训练到底在强化谁？
6. MCTS 访问计数分布为什么适合作为训练信号？在高平局任务里它会不会系统性失真？
7. Root-only GPU MCTS 是中间态还是正确终态？
8. 为什么 v1 目前主线选择 root-MCTS + tensor pipeline，而不是先做完整 GPU tree？
9. Child `value_only` 路径为什么语义上可接受？它会不会默默改变搜索分布？
10. 如果模拟次数无限增长，当前实现会收敛到更强策略，还是会被规则与评估噪声限制住？
11. Dirichlet noise 在这个项目里是必要探索，还是在给不稳定策略继续注入混乱？
12. 温度退火的最佳对象到底是动作采样多样性，还是 self-play 数据分布的课程调度？
13. 为什么 opening random moves 能改善训练，而不是单纯污染样本？
14. 当前 PUCT / root allocation 设计，是否真正符合这个游戏的分支结构？
15. 在阶段化动作游戏里，访问计数最多的动作是否真的就是最有价值的策略表达？

## 4. 强化学习目标

1. 这个项目到底在优化什么，是最终胜率、长期价值校准、局面排序能力，还是能稳定训练的代理目标？
2. `(s, π, z)` 目标在高平局率任务里是否足够？
3. 当绝大多数对局是平局时，value head 学到的到底是“局面无差别”，还是“训练信号过弱导致的塌缩”？
4. policy head 是否可能比 value head 更早学到有用信号？如果是，为什么最终还是容易陷入和棋偏好？
5. 当前 value bucket 设计比直接标量回归更好吗？好在表达能力、数值稳定性，还是 calibration？
6. bucket value 再映射成 WDL 的辅助目标，到底是在修正价值语义，还是在引入新的偏差？
7. 为什么 draw control 应优先从目标侧下手，而不是先修改规则或提高模拟次数？
8. anti-draw penalty 是在塑造更强策略，还是在用人为偏置把模型推出一个并不真实的最优点？
9. 如果加入中间奖励，究竟是在降低 credit assignment 难度，还是在破坏“只对最终胜负负责”的零和目标？
10. 在这个项目里，reward shaping 的风险主要是什么：作弊策略、局部最优、价值误校准，还是自博弈分布漂移？
11. replay window 应该追求更大还是更新鲜？
12. 旧样本在自博弈强化学习中到底是稳定器，还是历史包袱？
13. 如果训练集越来越大但棋力不升，根因更可能在样本质量、目标设计、优化器动态，还是评估口径失真？
14. 当前训练目标里，policy loss 与 value loss 的相对权重是否隐含了未被显式讨论的价值判断？
15. draw 样本在 policy loss 中应被弱化、保留，还是区别对待不同类型的和棋？
16. “和棋”在这个项目里是单一类别，还是其实混合了必和、可赢未赢、可输未输三种完全不同的学习情形？

## 5. 自博弈数据分布

1. 自博弈生成的数据，真的代表“应该学什么”，还是只代表“当前模型最常见地会做什么”？
2. 自博弈闭环为什么不会自动变强？它在哪些条件下反而会自我强化错误分布？
3. 当模型开始偏向和棋时，自博弈是否会把这种偏好固化成数据主流？
4. `decisive_game_ratio` 的提升，究竟意味着策略更强，还是只是局面更激进？
5. opening random moves 创造出来的对局，和真实强对局分布之间差距有多大？
6. 当前自博弈并发与分片策略，会不会隐含地改变样本独立性与温度行为？
7. staged pipeline 下的 shard 组织方式，会不会影响训练批次中的局面相关性？
8. 如果生成数据的模型和训练数据使用窗口重叠过大，会不会形成过强的自我蒸馏闭环？
9. 从“有效胜负样本增加”推到“模型更强”，中间缺了哪些必需论证？
10. 在高算力条件下，自博弈更可能受益于更多模拟，还是更多对局，还是更好的采样分布？

## 6. 优化与训练稳定性

1. 当前真正限制棋力增长的是算力、数据、优化器，还是目标定义？
2. 学习率调度为什么会影响棋力而不仅仅影响 loss 曲线？
3. 如果 loss 下降但强度不上升，说明是哪个环节出了问题？
4. DDP / streaming / staged load 这些工程优化，是否可能反过来影响训练统计性质？
5. 大 batch 在这里是稳定器还是伤害探索多样性的来源？
6. 更快的训练吞吐为什么不必然带来更强模型？
7. 当前 pipeline 中最可能引入“看不见的分布漂移”的环节是什么？
8. 非有限值、梯度爆炸、value collapse、policy entropy 异常之间，谁更像根因，谁更像症状？
9. 如果 value head 先崩，再拖垮 policy，日志上最早会出现什么征兆？
10. 当前模型容量是瓶颈吗？如果不是，为什么继续加深网络未必值得？

## 7. 评估与选模

1. 为什么 `vs_random` 高胜率不能证明模型真的强？
2. 如果 `vs_random` 和锦标赛排名不一致，到底谁更接近真实强度？
3. `wins > losses` 的 gating 为什么足够，而不是必须要求绝对胜率门槛？
4. 当前评估口径里，平局到底应该被看作中性结果、信息不足，还是明确的负面信号？
5. deterministic eval 和 sampled eval 分别在测什么？
6. 为什么 `vs_previous` 需要保留温度与采样，而 `vs_random` 默认走确定性评估？
7. Elo / BT 排名在高平局率环境下的解释力有多强？
8. 锦标赛冠军模型，是否一定适合作为下一轮训练基线？
9. “评估更严格”是否有可能系统性低估真实强度提升？
10. 如果一个模型更会赢随机对手，但更容易与强模型和棋，它到底是进步还是退步？
11. 评估中 opening random moves 的引入，是在测鲁棒性还是在污染比较条件？
12. 当前所有评估指标中，哪个最容易被工程实现细节误导？

## 8. 系统设计与工程取舍

1. 为什么当前主叙事聚焦 v1，而不是继续把 v0 做成唯一高性能主线？
2. v1 相对 v0 的真正价值，是吞吐提升、数据格式统一，还是未来可扩展性？
3. 为什么没有直接把所有逻辑都推到 C++/CUDA，而是保留了 Python 桥接与 staged 编排？
4. 什么时候“多一层抽象”是在帮项目，什么时候是在制造长期维护债务？
5. `v0_core` 作为底层能力复用边界，当前切分是否合理？
6. 为什么有些路径坚持与 v0 对齐，而有些又允许 v1 自己形成新语义？
7. 训练主入口为什么应该固定在 `scripts/big_train_v1.sh`，而不是把全部能力塞进一个更通用但更复杂的统一 CLI？
8. 当前文档结构强调 v1、弱化 legacy/v0 细节，这会提升理解效率，还是掩盖关键历史背景？
9. 如果未来需要支持规则变体，当前文档与代码结构的哪些部分会先崩？
10. “保持设计简洁、不增加开关”这条原则，在强化学习系统里什么时候会变成障碍？

## 9. 面向研究的问题

1. 六洲棋这个任务的最优研究方向，真的是继续堆 AlphaZero 范式吗？
2. 是否应该引入更强先验，例如结构特征、局面模板、价值分解或课程学习？
3. 当前项目里最值得做论文式分析的问题是什么：高平局率、动作阶段化、价值校准，还是 GPU 自博弈系统设计？
4. 这个任务更像一个“规则复杂但状态空间有限”的系统，还是一个“信用分配极差”的系统？
5. 如果把 MCTS 去掉，用更大的 policy/value 模型直接做策略迭代，效果会怎样？
6. 如果引入 reanalyse、离线训练或专家规则引导，会不会比继续闭环自博弈更划算？
7. 当前模型学到的是“六洲棋知识”，还是“当前训练制度下的生存策略”？
8. 项目最终想证明的到底是什么：能做强 agent，能做高性能系统，还是能把一个复杂民间棋类形式化并训练起来？

## 10. 给未来自己的追问

1. 你现在相信的结论，究竟来自代码事实、实验结果，还是来自你希望它成立？
2. 你有没有把“训练更快”误当成“方法更对”？
3. 你有没有把“指标变好”误当成“真实棋力变强”？
4. 你有没有把“当前实现方便”误当成“长期设计合理”？
5. 如果今天必须删掉一半复杂度，你会删掉哪一半？为什么？
6. 如果今天必须保留一个最核心的研究问题继续做三个月，那会是哪一个？
7. 如果未来证明当前很多设计都不是最优，你希望项目里哪一部分仍然是值得保留的？

## 使用建议

- 不要急着回答所有问题。
- 先挑那些最能决定项目方向的问题。
- 当你的答案改变时，说明你对系统的理解正在变深，而不是说明前面白想了。
- 如果你准备把这些问题逐步回答下来，建议后续单独建立一份“问题回答手册”或研究日志，与本页分离维护。
