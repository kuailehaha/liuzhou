# 人机对战系统

除了训练流水线外，本项目还提供了一个完整的人机对战系统，便于做模型试玩、规则验证和基础演示。

## 1. 组成

### 后端

后端位于 `backend/`，使用 FastAPI 提供接口。

核心职责：

- 创建新对局
- 维护会话状态
- 校验玩家动作
- 调用 AI 选择动作
- 返回棋盘状态、合法动作和评估信息

主入口：`backend/main.py`

### 前端

前端位于 `web_ui/`，提供棋盘渲染、交互和状态展示。

主页面由后端挂载到：

- `/ui/index.html`

## 2. 请求流

一次典型对局交互流程如下：

1. 用户在前端选择执子方、模型路径和搜索参数
2. 前端向后端发起新建对局请求
3. 后端创建 `GameSession`
4. 若轮到 AI，后端调用 MCTS agent 自动走子
5. 每次落子后，后端返回最新棋盘、合法动作和附加信息
6. 前端刷新棋盘、状态栏和动作记录

## 3. 模型加载逻辑

后端会尝试按给定路径加载模型。

相关默认配置位于 `backend/main.py`：

- `MODEL_PATH`
- `MODEL_DEVICE`
- `MCTS_SIMULATIONS`
- `MCTS_TEMPERATURE`

如果没有成功加载模型，系统可以退回到 `RandomAgent`，以保证试玩路径仍然可用。

## 4. 评估信息

后端不仅负责执行 AI 动作，也会尝试读取价值头输出，给出当前局面的估计信息。

这部分信息适合用于：

- 前端展示胜率或局面价值
- 快速观察模型对局面变化的反应
- 排查模型是否明显偏向某类状态

## 5. 适用场景

人机对战系统主要适合三类场景：

- 展示项目能力
- 手工验证规则与阶段流转
- 用小规模搜索直接体验 checkpoint 的下棋风格

它不是大规模评估工具；真正的模型比较仍应走 `scripts/eval_checkpoint.py` 或 `scripts/tournament_v1_eval.py`。
